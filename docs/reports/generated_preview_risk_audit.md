# Generated Preview Risk Audit

## 개요

이 문서는 `.ai OS` Phase 8 generated preview contract의 위험 모델을 정의한다. Generated preview는 future generated output을 current target state와 비교하기 위한 read-only input이다. 그러나 preview와 adapter generation, preview와 sync apply의 경계가 흐려지면 mutation 안전성이 약해질 수 있다.

이번 감사는 위험 분석만 수행한다. 런타임 코드, `.ai` rules, adapter generation, generated content creation, sync apply, manifest persistence는 변경하지 않는다.

## Risk Matrix

| --- | --- | --- | --- | --- |

| over-broad activation | Medium | Medium | activation/rule context가 과도하게 넓으면 preview content에 불필요한 규칙이 섞일 수 있다. | activation reference는 optional로 두고, unresolved 상태는 `activation-unresolved` unavailable reason으로 처리한다. |
| nondeterministic adapter output | High | Medium | 같은 input이 매번 다른 generated hash를 만들면 dry-run update 판단이 불안정해진다. | adapter identity/version과 deterministic metadata를 요구하고, deterministic output 전까지 update candidate를 만들지 않는다. |

## 주요 위험 상세

### Stale Source

Generated preview는 source content에서 생성된다. source가 변경되었는데 manifest `source_hash`가 갱신되지 않았다면 preview는 현재 repository state와 manifest state 중 어느 쪽을 기준으로 했는지 불명확해진다.

정책:

- dry-run은 source hash mismatch를 preview generation보다 먼저 평가해야 한다.
- stale source가 감지되면 preview comparison은 update candidate를 만들면 안 된다.
- preview provenance에는 source path와 source hash를 반드시 보존한다.

### Over-broad Activation

Activation v1은 future sync planning input에 영향을 줄 수 있지만, 현재 sync runtime은 activation-driven selection을 구현하지 않는다. Preview가 activation context를 과도하게 가져오면 생성 결과가 불안정해지고 noisy해질 수 있다.

정책:

- activation/rule context reference는 optional이다.
- activation을 resolve할 수 없으면 `activation-unresolved`로 preview unavailable 처리한다.
- activation은 write authorization, ownership 변경, drift-stop 우회에 사용할 수 없다.

### Nondeterministic Adapter Output

Adapter output이 deterministic하지 않으면 generated hash가 안정적인 contract가 될 수 없다. 이는 dry-run output contract와 snapshot test를 약화시킨다.

정책:

- adapter identity와 version을 generated metadata에 기록한다.
- same input + same adapter version은 same generated bytes hash를 만들어야 한다.
- deterministic guarantee가 없으면 update candidate를 만들지 않는다.

### False Update Candidate

Generated hash가 current target hash와 다르다고 해서 항상 안전한 update 후보는 아니다. Target이 이미 drifted 상태이면 generated hash 비교보다 drift-stop이 우선해야 한다.

정책:

- target이 manifest `target_hash`와 clean하게 일치할 때만 generated hash 비교를 수행한다.
- drifted target은 generated preview가 있더라도 `drift-stop`이다.
- invalid marker는 generated preview가 있더라도 `conflict`다.
- preview unavailable은 update candidate를 만들지 않는다.

### Line Ending Drift

Phase 7 v0 hash policy는 observed UTF-8 bytes를 사용하고 CRLF/LF를 정규화하지 않는다. Preview provider가 임의로 line ending을 정규화하면 false drift 또는 false update가 생길 수 있다.

정책:

- preview generated bytes는 Phase 7 v0 hash policy를 따른다.
- generated preview는 line ending, trailing newline, whitespace를 보존해야 한다.
- future normalization은 별도 hash policy version 없이는 도입하지 않는다.

### Generated Content Noise

Generated content가 docs/reports/plans 또는 broad context를 무분별하게 섞으면 runtime target에 불필요한 설명, 감사 내용, 계획 내용이 포함될 수 있다.

정책:

- preview provenance는 source paths와 context reference를 남긴다.
- docs artifacts는 runtime input이 아니며 명시 selection 없이는 preview source가 될 수 없다.
- preview provider는 source selection boundary를 출력 metadata에 남겨야 한다.

### Preview/Apply Boundary Confusion

Preview는 "쓸 수 있는 결과"처럼 보이기 쉽다. 그러나 Phase 8 초입에서 preview는 비교 계약일 뿐이고 mutation gate를 열지 않는다.

정책:

- preview result는 sync apply authorization이 아니다.
- `update` candidate는 informational planning result다.
- mutation은 transaction/rollback/marker write policy/readiness gate 이후에만 가능하다.

## Unavailable State Risk

Preview unavailable은 실패일 수도 있고 단순 정보일 수도 있다. unavailable reason을 숨기면 사용자는 dry-run이 update candidate를 판단하지 못한 이유를 알 수 없다.

Required unavailable reasons:

- `adapter-unavailable`
- `source-missing`
- `unsupported-sync-mode`
- `marker-invalid`
- `generation-disabled`
- `activation-unresolved`

Severity guidance:

- `source-missing`: 기존 blocking conflict가 우선한다.
- `marker-invalid`: 기존 marker conflict가 우선한다.
- `adapter-unavailable`: info 또는 warning, update candidate 없음
- `generation-disabled`: info, update candidate 없음
- `unsupported-sync-mode`: warning, update candidate 없음
- `activation-unresolved`: warning, update candidate 없음

## Envelope and Reporting Risk

Envelope v2가 preview detail을 잃으면 downstream tooling이 update candidate의 근거를 검증할 수 없다.

정책:

- generated hashes는 `payload.results[].hashes`에 보존한다.
- preview unavailable reason은 `payload.results[].details` 또는 `messages[]`에 보존한다.
- provenance는 `payload.results[].details.preview.provenance` 또는 `meta.preview`에 보존한다.
- preview warning/info는 command status를 불필요하게 fail로 만들면 안 된다.

## Mutation Boundary Risk

Generated preview가 생기면 sync apply에 가까워 보이지만, 다음 항목은 계속 금지해야 한다.

- target write
- manifest write
- transaction log write
- marker insertion
- marker repair
- marker deletion
- adapter generation
- rollback execution
- force
- decommission

Mutation readiness는 별도 audit과 explicit gate가 필요하다.

## Future Implementation Risk Controls

Future implementation은 다음 순서로만 진행하는 것이 안전하다.

1. Preview fixture contract
   - input/output fixture를 먼저 만들고 provider 없이 schema를 고정한다.

2. Fixture-backed read-only provider
   - deterministic fixture provider로 comparison behavior를 검증한다.
   - adapter execution은 여전히 금지한다.

3. Dry-run update candidate extension
   - clean target에서만 generated hash comparison을 수행한다.
   - preview unavailable은 기존 Phase 7 behavior를 유지한다.

4. Adapter planning
   - deterministic adapter output contract가 고정된 뒤 별도 계획으로 진행한다.

5. Mutation readiness
   - transaction, rollback, marker write, decommission policy가 끝난 뒤에만 검토한다.

## 결론

Generated preview는 Phase 7 read-only evaluator를 다음 단계로 확장하기 위한 적절한 Phase 8 시작점이다. 그러나 preview는 생성물 비교 계약일 뿐 mutation 권한이 아니다. 가장 큰 위험은 stale source, nondeterministic output, false update candidate, preview/apply boundary confusion이다. 이 위험은 source hash precedence, deterministic metadata, clean-target-only comparison, explicit mutation block으로 완화해야 한다.
