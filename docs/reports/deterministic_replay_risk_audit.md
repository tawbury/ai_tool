# Deterministic replay 위험 감사

## 개요

이 문서는 future real preview provider의 deterministic replay test architecture에 대한 위험을 감사한다. Replay test는 provider 구현 이후 output stability를 검증하기 위한 핵심 gate이며, 잘못 설계되면 nondeterminism을 숨기거나 false update candidate를 허용할 수 있다.

이번 감사는 문서 작업만 수행한다. Provider 구현, adapter runtime, generated content generation, sync apply, mutation은 구현하지 않는다.

## 주요 위험 상세

### Replay mismatch retry masking

Replay mismatch가 발생했을 때 자동 retry를 수행하면 nondeterministic provider가 우연히 snapshot과 일치할 수 있다. 이는 replay test의 목적을 훼손한다.

완화:

- Replay mismatch는 즉시 validation failure로 처리한다.
- Automatic retry를 금지한다.
- Timeout은 `provider-timeout`, nondeterminism은 `nondeterministic-output`으로 분류한다.

### Incomplete comparison

Generated hash만 비교하면 provider metadata나 provenance drift를 놓칠 수 있다. 동일 hash라도 source provenance가 바뀌었다면 dry-run 설명력은 손상된다.

완화:

- generated hash fields 모두 비교한다.
- provenance 전체를 비교한다.
- provider metadata 전체를 비교한다.
- deterministic flag와 unavailable reason도 비교한다.

### Provider version drift hidden

Provider output logic이 바뀌었는데 `provider_version`이 유지되면 replay snapshot의 의미가 깨진다.

완화:

- 같은 provider version에서는 exact output match를 요구한다.
- Output-affecting change는 provider version bump를 요구한다.
- Version bump가 있으면 snapshot 갱신과 migration note를 요구한다.

### Line ending normalization leak

Replay framework가 JSON fixture를 읽는 과정에서 source bytes나 generated bytes 의미를 정규화하면 실제 hash policy 위반을 탐지하지 못한다.

완화:

- Comparison은 hash string exact match를 사용한다.
- Hash 계산 정책은 observed UTF-8 bytes를 유지한다.
- LF, CRLF, trailing newline, BOM case를 별도 replay dimension으로 둔다.

### Unavailable reason instability

동일 unavailable input이 어떤 때는 `adapter-unavailable`, 어떤 때는 `provider-timeout`으로 나오면 downstream dry-run 설명이 불안정해진다.

완화:

- Unavailable reason은 exact match 대상이다.
- Provider failure classification order를 명시한다.
- `nondeterministic-output`과 `provider-timeout`을 별도 reason으로 유지한다.

### Provenance ordering instability

Source paths가 set이나 filesystem traversal 순서에 따라 달라지면 replay mismatch가 발생하거나, 더 나쁘게는 output이 흔들린다.

완화:

- Source path order를 input fixture에서 명시한다.
- Provider는 input order를 보존하거나 명시 정렬 정책을 사용한다.
- Provenance comparison은 exact order comparison을 기본값으로 한다.

### Provider metadata under-specified

Provider metadata가 충분하지 않으면 replay output이 어떤 provider/version/hash policy에서 만들어졌는지 알 수 없다.

완화:

- Provider snapshot에 provider id, version, deterministic capability, supported sync modes, hash policy를 포함한다.
- Output metadata에도 provider id/version/hash policy를 보존한다.
- Missing provider metadata는 replay validation error다.

### Fixture staleness

Replay fixture가 provider contract 변경을 따라가지 못하면 테스트가 오래된 계약을 검증할 수 있다.

완화:

- Replay manifest schema version을 둔다.
- Fixture inventory validation을 수행한다.
- Contract 변경 시 fixture migration note를 남긴다.
- Snapshot auto-update는 금지한다.

### False update confidence

Replay test를 통과했다는 이유만으로 dry-run update 후보가 안전하다고 판단하면 안 된다. Replay는 provider stability만 검증한다.

완화:

- Dry-run은 manifest/source/marker/drift checks를 계속 우선한다.
- Replay success는 provider output 사용 가능성만 의미한다.
- Mutation/apply readiness와는 무관하다고 명시한다.

### Unavailable replay treated as pass-to-update

Unavailable output replay가 성공했다는 이유로 update 후보가 만들어지면 위험하다.

완화:

- Unavailable output은 generated hash fields가 모두 `null`이어야 한다.
- `preview_available: false`는 no-update를 의미한다.
- Replay validation은 unavailable output의 안정성만 검증한다.

### BOM and trailing newline omission

BOM과 trailing newline case가 없으면 provider가 bytes policy를 위반해도 replay suite가 잡지 못할 수 있다.

완화:

- BOM preserved case를 fixture dimension에 포함한다.
- trailing newline present/absent case를 포함한다.
- LF/CRLF semantic-same-but-bytes-different case를 포함한다.

### External state dependency

Provider가 network, wall-clock time, environment variables, random seeds, filesystem traversal order에 의존하면 replay가 불안정해진다.

완화:

- Provider input에 없는 external state 의존을 금지한다.
- External dependency가 필요한 provider는 deterministic capable이 될 수 없다.
- External dependency failure는 unavailable output으로 분류한다.

## Replay failure classification

권장 replay failure code:

- `replay-hash-mismatch`
- `replay-provenance-mismatch`
- `replay-provider-metadata-mismatch`
- `replay-unavailable-reason-mismatch`
- `replay-deterministic-flag-mismatch`
- `replay-schema-mismatch`
- `replay-fixture-missing`
- `replay-provider-version-drift`

Failure severity:

- Replay mismatch는 validation failure다.
- Fixture inventory issue도 validation failure다.
- Provider version changed with explicit snapshot migration은 failure가 아닐 수 있지만 migration note가 필요하다.

## Read-only boundary

Replay test는 read-only여야 한다.

허용:

- fixture read
- provider input/output schema validation
- hash string comparison
- metadata/provenance comparison
- future explicitly approved provider validation execution

금지:

- snapshot auto-update
- target file write
- manifest write
- transaction log write
- generated content persistence
- marker insertion, repair, deletion
- sync apply
- rollback

## Future validation checklist

Replay validation은 다음을 확인해야 한다.

- Replay manifest schema is valid.
- Provider snapshot exists.
- Provider id/version match replay manifest.
- Every case has input and expected output fixtures.
- Input fixture schema is valid.
- Expected output fixture schema is valid.
- Non-null hashes match `sha256:<lowercase-hex>`.
- Actual output matches expected generated hashes.
- Actual output matches expected provenance.
- Actual output matches expected provider metadata.
- Actual output matches expected unavailable reason.
- Actual output matches expected deterministic flag.
- Unavailable outputs have null generated hash fields.

## 결론

Deterministic replay test architecture의 핵심 위험은 mismatch를 숨기는 retry, 불완전한 비교, provider version drift, bytes policy 누락이다. Replay validation은 hash뿐 아니라 provenance, metadata, unavailable reason, deterministic flag를 exact match로 비교해야 한다. Replay 성공은 provider stability를 의미할 뿐이며 mutation이나 sync apply를 승인하지 않는다.
